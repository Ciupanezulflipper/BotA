# BotA Provider Accounting and Pipeline Ledger Repair

Date: 2026-07-22

## Executive finding

The Telegram warning labelled `Twelve Data credits: 600/800` was generated by a local generic counter, not by a Twelve Data account-usage response.

The live fetch path is OANDA primary with Yahoo fallback. The updater called the generic counter after every successful fetch. With the default updater scope of five pairs and four timeframes, that is up to 20 generic increments per updater cycle. Thirty successful cycles therefore produce the displayed value 600 even if Twelve Data was never called.

This repair removes the false equivalence between successful market-data fetches and Twelve Data credits.

## Scope lock

Unchanged:

- live pairs and timeframes;
- scoring and quality filters;
- ADX gate;
- Telegram score/tier thresholds;
- SL/TP and risk logic;
- signal dedup semantics;
- Supabase signal schema and strategy behavior.

## Implemented controls

### Provider-specific request accounting

`tools/provider_usage.py` records the actual provider and request result in:

- `logs/provider_calls.jsonl` — append-only request ledger;
- `state/provider_usage.json` — compact daily counters.

OANDA and Yahoo requests always consume zero Twelve Data credits.

A future Twelve Data caller must reserve credits before its request. Default budget:

- daily limit: 800;
- protected reserve: 100;
- usable hard cap: 700.

Reservations are serialized with an exclusive file lock, so concurrent callers cannot oversubscribe the budget.

`tools/api_credit_tracker.py increment` now fails closed instead of accepting an unidentified generic increment.

### Network-boundary evidence

`tools/data_fetch_candles.sh` records each actual request attempt:

- OANDA success or failure;
- Yahoo success, failure, or HTTP 429 block.

This captures failed OANDA attempts before Yahoo fallback and avoids updater-level double counting.

### Useful-progress ledger

`tools/pipeline_ledger.py` writes:

- `logs/pipeline_events.jsonl` — append-only component and decision events;
- `state/pipeline_progress.json` — latest current-boot progress.

Events include boot ID and `CLOCK_BOOTTIME` monotonic time. Same-boot freshness does not depend on Android wall-clock changes.

### Per-cycle decision reconciliation

`tools/run_signal_watcher_with_ledger.sh` captures the pre-cycle byte offsets of:

- `logs/alerts.csv`;
- `logs/cron.signals.log`.

`tools/watcher_cycle_ledger.py` reads only bytes appended during that cycle and records a terminal result for EURUSD/M15 and GBPUSD/M15. Historical content cannot be counted as current evidence.

Terminal outcomes distinguish stale/missing cache, pause/news/calendar gates, parse failures, filter rejection, score/tier gates, cooldown, dedup, Telegram delivery, and missing terminal evidence.

### Exact runtime health

`tools/control_plane_status.py` requires:

- exactly one standard runsvdir manager;
- all seven required supervisors owned by that manager;
- zero PID 1 orphans;
- seven running services;
- one live `crond -n -s` matching the current crond supervise PID.

`tools/pipeline_health.py` requires current-boot monotonic progress and current EURUSD/M15 plus GBPUSD/M15 terminal decisions while the market is open.

A `started` event has only a short grace period. It cannot remain healthy indefinitely.

`tools/bota_supervisor.sh` is read-only toward services. It no longer starts detached crond. Recovery is emitted only when exact ownership, useful progress, and clock safety all pass.

### Honest heartbeat

`tools/heartbeat.sh` now reports Telegram reachability separately from signal health. Its message includes the current runtime mode, ownership, orphan count, and useful-progress status. It no longer owns an independent wall-clock recovery/dead-man state machine.

## Runtime rollout boundary

Repository implementation does not itself change the phone runtime.

Before deployment:

1. CI must pass.
2. The pull request must be reviewed and merged.
3. The phone control plane must be stabilized to one manager, seven owned supervisors, zero orphans, and one supervised crond.
4. The active watcher runit script must be inspected to identify its exact watcher invocation.
5. A separate staged phone package must back up the active run scripts and change only the watcher invocation to `tools/run_signal_watcher_with_ledger.sh`.
6. Execution requires a fresh boot-bound typed approval.
7. Post-deployment verification must prove provider counters, two terminal decisions per open-market watcher cycle, exact ownership, and honest health transitions.

## Rollback

Repository rollback is a revert of this pull request.

Phone rollback must restore the backed-up watcher run script and the pre-deployment repository commit, then independently verify one manager, seven owned supervisors, zero orphans, and one supervised crond.
